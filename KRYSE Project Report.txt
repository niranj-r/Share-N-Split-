2. FIREBASE DATABASE ARCHITECTURE & SCHEMAS
--------------------------------------------------------------------------------
The backend database is built on Google Cloud Firebase Firestore, a NoSQL document 
database. The data model is structured around users, budgets, trips, expenses, 
groups, and shared expenses, with document IDs linking entities together.

A. Users Collection ('users')
   - Path: /users/{uid}
   - Description: Stores profile and preferences for registered users.
   - Fields:
     * name: String (Full name of user)
     * email: String (Email address used for logging in and group splitting)
     * preferences: Map
       - theme: String ("dark" | "light")
     * createdAt: Timestamp (Firestore Server Timestamp)

B. Budgets Collection ('budgets')
   - Path: /budgets/{budgetId}
   - Description: Pre-declared financial thresholds for specific timeframes.
   - Fields:
     * budgetId: String (Auto-generated Document ID)
     * userId: String (User UID of owner - links to /users/{uid})
     * name: String (Budget name, e.g. "Bali Trip Budget", "Monthly Groceries")
     * totalAmount: String/Number (Total allocated funds)
     * amountSpent: Number (Default 0, modified dynamically or computed on read)
     * startDate: String (YYYY-MM-DD format)
     * endDate: String (YYYY-MM-DD format)
     * createdAt: Timestamp (Firestore Server Timestamp)

C. Trips Collection ('trips')
   - Path: /trips/{tripId}
   - Description: Travel itineraries mapped to budgets.
   - Fields:
     * tripId: String (Auto-generated Document ID)
     * name: String (Name of the trip, e.g. "Goa Expedition")
     * destination: String (Geographic location)
     * startDate: String (YYYY-MM-DD format)
     * endDate: String (YYYY-MM-DD format)
     * budgetId: String (FK, refers to /budgets/{budgetId}, optional)
     * createdBy: String (User UID of trip creator)
     * createdAt: Timestamp (Firestore Server Timestamp)

D. Expenses Collection ('expenses')
   - Path: /expenses/{expenseId}
   - Description: Personal, individual transactions.
   - Fields:
     * expenseId: String (Auto-generated Document ID)
     * userId: String (User UID of owner)
     * title: String (Description of transaction, e.g. "Airport Taxi")
     * amount: String/Number (Money spent)
     * category: String (Food, Travel, Shopping, Bills, Entertainment, Others)
     * date: String (YYYY-MM-DD format)
     * budgetId: String (FK, refers to /budgets/{budgetId}, optional)
     * tripId: String (FK, refers to /trips/{tripId}, optional)
     * notes: String (User notes, optional)
     * createdAt: Timestamp (Firestore Server Timestamp)

E. Groups Collection ('groups')
   - Path: /groups/{groupId}
   - Description: Collaborators sharing expenses (e.g. travel roommates).
   - Fields:
     * groupId: String (Auto-generated Document ID)
     * name: String (Group name, e.g. "Flat 404", "EuroTrip 2026")
     * description: String (Group description, optional)
     * members: Array of Strings (List of participant emails - used for querying)
     * createdBy: String (User UID of creator)
     * createdAt: Timestamp (Firestore Server Timestamp)

F. Shared Expenses Collection ('sharedExpenses')
   - Path: /sharedExpenses/{sharedExpenseId}
   - Description: Split expenses belonging to a group.
   - Fields:
     * sharedExpenseId: String (Auto-generated Document ID)
     * groupId: String (FK, refers to /groups/{groupId})
     * description: String (Description of shared transaction)
     * amount: Number (Total expense amount)
     * date: String (YYYY-MM-DD format)
     * budgetId: String (FK, refers to /budgets/{budgetId}, optional)
     * splitType: String ("EQUAL" | "CUSTOM" | "PERCENTAGE" | "SHARES")
     * payers: Map (key: User Email -> value: Amount Paid, allows multi-payer)
     * splits: Map (key: User Email -> value: Calculated Amount Owed)
     * members: Array of Strings (List of participant emails - duplicate of group members)
     * paidBackBy: Array of Strings (Emails of users who settled their debt)
     * createdAt: Timestamp (Firestore Server Timestamp)

     * Right Column: Shared expenses log and a "Settlement Summary" box.
   - Settlement Summary Panel: Calculates net balances and displays minimized transactions

I. Settings Page

4. SYSTEM LOGIC & DATA INTERCONNECTIONS
--------------------------------------------------------------------------------
The backend and frontend are highly interconnected through modular hooks and active 
listeners that dynamically sync and propagate data.

A. Authentication & State Management
   - Handled via context/AuthContext.js. Renders a React context provider wrapping 
     the application tree.
   - Holds currentUser state. On state change, it subscribes to collections.

B. Real-time Subscriptions (onSnapshot)
   - When a screen loads, services.js establishes real-time Firestore listeners.
   - Any CRUD operations perform a write to Firestore. The listener automatically 
     receives the document snapshot, updating the React state and forcing a re-render.
   - All budget rings, charts, and values update automatically across all devices.

C. The "Virtual Expense" Integration
   - Shared expenses are stored globally. The system queries sharedExpenses with:
     where('members', 'array-contains', currentUser.email)
   - On fetch, the application calculates:
     virtualAmount = sharedExpense.splits[currentUser.email]
   - It maps this into a virtual expense object prefixed with "[Shared]" and merges 
     it with personal expenses. This ensures that a user's dashboard charts, recent 
     expenses list, and budget utilization correctly reflect their shared liabilities.

D. Multi-Payer Split Calculations
   - When adding a shared expense, the total amount must match the sum of payer contributions:
     totalAmount == SUM(expForm.payers[member])
   - The user selects a split method, which determines how much each member owes:
     * EQUAL: splits = totalAmount / count (handling remainders to ensure exact sum).
     * PERCENTAGE: splits = (totalAmount * percentage[member]) / 100
     * SHARES: splits = (totalAmount * share[member]) / totalShares
     * CUSTOM: splits = exactAmount[member]
   - The system records both payers and splits in Firestore.

E. ShareNSplit Greedy Settlement Optimization Algorithm
   To prevent complex round-robin transfers (e.g. A owes B, B owes C), the system 
   contains an optimization algorithm (optimizeSettlements in splitUtils.js):
   1. Compute Net Balances:
      For each group member: Balance = Total Paid - Total Owed.
      * Total Paid = Sum of all values where member is key in expense 'payers'.
      * Total Owed = Sum of all values where member is key in expense 'splits'.
   2. Separate Members:
      - Creditors: Balance > 0.01
      - Debtors: Balance < -0.01 (expressed as absolute value)
   3. Sort Collections:
      Sort creditors descending and debtors descending.
   4. Greedy Matching:
      - Match the largest debtor with the largest creditor.
      - Calculate transaction amount: Transferred = min(creditor.amount, debtor.amount).
      - Record transaction: Debtor pays Creditor (Transferred).
      - Adjust balances. If a balance drops to 0, advance to the next member.
      - Repeat until all debts are settled, resulting in the absolute minimum transactions.

F. Cascade Deletion Flow
   - Deleting a budget or trip retains individual expenses but removes references.
   - Deleting a group queries and deletes all sharedExpenses belonging to that group, 
     then removes the group itself. This cascade logic avoids database orphans.

5. ANIMATION SYSTEM (Implemented & Planned)
--------------------------------------------------------------------------------

- Page Transitions: Smooth opacity + Y-axis slide (0.3s) when swapping routes.

- Logo Assembly: 3 triangles assembling on login screen load (GSAP).
- Number Ticker: Budget values "counting up" when the dashboard loads.
- Success Draw: SVG path animation for checkmarks after adding an expense.
- Pulse Warning: Budget rings should "pulse" orange when spending > 90%.
